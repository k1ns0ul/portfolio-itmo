# Генератор отчетов по крипто-портфелям

## Что это

На вход — адрес кошелька. На выходе — связный аналитический отчет: состав портфеля, PnL, диверсификация, риск-скор. Текст пишет локальная LLM, дообученная под формат финансовых отчетов. Никаких внешних API: все крутится на своих серверах, данные не уходят наружу.

## Как устроено

Go API на Gin принимает запрос с адресом кошелька. Из ClickHouse тянутся транзакции и текущие балансы — используются те же таблицы, что и в проекте `02-wallet-scoring`. Go считает метрики:

- PnL (реализованный и нереализованный) по каждому токену и в сумме
- доля стейблкоинов в портфеле
- Herfindahl index для оценки диверсификации
- risk score на основе волатильности позиций и доли скам-/подозрительных токенов
- топ позиций по объему

Метрики собираются в structured JSON и уходят в Python LLM Service. Там Llama 3 8B (fine-tuned через QLoRA) превращает цифры в текст: короткое резюме, разбор позиций, отметки рисков, рекомендации. Go API получает результат, объединяет с метриками и отдает клиенту JSON со structured-полями и текстом отчета.

Инференс LLM крутится через vLLM (для GPU) или llama.cpp (если GPU нет). Конкретный backend выбирается переменной окружения.

## ML

**Датасет.** Пары `JSON-метрики → текст отчета`. Минимум 500 примеров, собирались полуавтоматически: метрики реальные, тексты сначала писались вручную и через инструктивные промпты к большой модели, потом редактировались.

**Fine-tuning.** QLoRA на Llama 3 8B. 4-bit квантизация базовой модели, LoRA-адаптеры на attention и MLP. Тренировка на одной потребительской GPU (24 ГБ хватает). Метрики качества — BLEU и ROUGE на отложенной выборке плюс ручная оценка адекватности фактов.

**Инференс.** vLLM держит модель в VRAM и батчит запросы. Для CPU-only варианта собирается GGUF и поднимается через llama.cpp с тем же HTTP-интерфейсом. Системный промпт фиксирует формат отчета и ограничивает длину.

## Запуск

```bash
docker-compose up -d
```

Переменные:

| Переменная | Описание |
|------------|----------|
| `CLICKHOUSE_DSN` | DSN ClickHouse с данными из проекта 01 |
| `REDIS_ADDR` | `redis:6379` |
| `LLM_BACKEND` | `vllm` или `llamacpp` |
| `LLM_MODEL_PATH` | путь к весам внутри контейнера |
| `LLM_SERVICE_URL` | внутренний URL Python-сервиса |
| `API_PORT` | по умолчанию `8080` |

## Структура проекта

```
.
├── cmd/
│   └── api/             # HTTP API на Gin
├── internal/
│   ├── clickhouse/      # запросы к таблицам из проекта 01
│   ├── metrics/         # PnL, Herfindahl, risk score
│   ├── llm/             # клиент к Python-сервису

├── ml/
│   ├── data/            # датасет JSON → текст
│   ├── train_qlora.py   # fine-tuning
│   ├── serve_vllm.py    # инференс через vLLM
│   ├── serve_llamacpp.py
│   └── model/           # адаптеры и GGUF
```
