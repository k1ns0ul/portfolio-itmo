# Т-Выгода

## Что это

Бэкенд платформы лояльности, спроектированный как кейс для банковского приложения. Пользователи публикуют посты о выгодных покупках: фото товара, цена до и после скидки, ссылка на промокод партнера. За приведенных друзей капают бонусы. Между банком и партнерами идут CFA-расчеты: каждое погашение промокода фиксируется как цифровое денежное требование по логике 259-ФЗ.

## Бизнес-логика

- **Посты.** Пользователь публикует находку, прикрепляет промокод партнера. Лента ранжируется рекомендательной моделью.
- **Промокоды.** Привязаны к партнеру и CPA-комиссии. Активация фиксируется в Kafka, дальше события расходятся по подписчикам.
- **Рефералы.** Три уровня: за приглашенного друга 10% от CPA, за друга друга 3%, за третий уровень 1%. Цепочки хранятся в PostgreSQL, чтобы при `purchase.confirmed` сразу начислять бонусы вверх по дереву.
- **CFA-расчеты.** При каждом подтвержденном погашении промокода создается settlement-запись: банк должен партнеру или наоборот. Балансы сводятся периодически.

## Как устроено

API на Go и Gin, БД — PostgreSQL. Kafka разносит события между сервисами, основные топики:

- `promo.activated` — пользователь применил промокод
- `purchase.confirmed` — партнер подтвердил покупку
- `referral.credited` — бонус начислен реферралу
- `cfa.settlement.created` — создано CFA-требование

Redis держит горячие данные: топы покупателей, кэш персональных рекомендаций, серии заходов (стрики).

В PostgreSQL основные таблицы: `users`, `posts`, `promos`, `partners`, `referral_chains`, `cfa_balances`, `cfa_settlements`.

## ML

**Рекомендательная модель.** Collaborative filtering на матрице `user × category`. Поверх — контекстные фичи: время суток, день недели, сезонность, история кликов и покупок пользователя. Обновляется батчами раз в час, инференс через отдельный Python-сервис.

**Антифрод.** Ищет аномалии в графе рефералов. Признаки: кольца (A пригласил B, B пригласил A через посредника), массовые регистрации с одного устройства или одной подсети, неестественные временные паттерны активаций. На графе крутится community detection, подозрительные кластера улетают на ручную проверку.

## CFA-модуль

При событии `purchase.confirmed` Go-воркер создает запись в `cfa_settlements`: кто должник, кто кредитор, сумма, идентификатор базового погашения, статус. Это упрощенная модель цифрового денежного требования. Не блокчейн и без эмиссии токена, но структура данных и event log соответствуют принципам учета по 259-ФЗ: неизменяемый журнал событий, привязка к первичному факту, явные стороны обязательства.

Settlement переходит по статусам `created → confirmed → settled`. Закрытие баланса между банком и партнером — отдельная процедура с актом сверки.

## Запуск

```bash
docker-compose up -d
```

Переменные:

| Переменная | Описание |
|------------|----------|
| `POSTGRES_DSN` | `postgres://app:app@postgres:5432/tvygoda` |
| `KAFKA_BROKERS` | `kafka:9092` |
| `REDIS_ADDR` | `redis:6379` |
| `API_PORT` | по умолчанию `8080` |
| `JWT_SECRET` | секрет для подписи токенов |
